# RegexMaker Project - Development Guidelines

## Architecture & Patterns

### MVVM Pattern
- **STRICTLY follow MVVM pattern** for all new code
- ViewModels should be instantiated in XAML, not code-behind
- Use data binding instead of direct control manipulation where possible
- Use `ReflectionBinding` for dynamic ViewModel types (e.g., node parameter panels)

### Acceptable Code-Behind Scenarios
The following scenarios are **acceptable** for code-behind due to platform limitations or complexity:

1. **Drag-Drop Handlers** - Pointer event handling for custom drag-drop (e.g., toolbox items)
2. **Canvas Manipulation** - Direct manipulation of `DragCanvas.Children` collection
3. **Platform APIs** - File dialogs, storage APIs that require code-behind interaction
4. **Custom Control Internals** - Visual tree creation within custom control constructors (e.g., `RgxNodeControl`)

### Critical: Do Not Modify Working Components
- **SpToolBox**: Dynamic TextBlock creation works perfectly - DO NOT attempt to convert to ItemsControl or other MVVM approaches
- **Drag-drop functionality**: Keep existing pointer event handlers - they work reliably
- **DragCanvas operations**: Direct Children collection manipulation is necessary and correct

## Code Style

### Avalonia/XAML
- Use compiled bindings with `x:DataType` where type is known at compile time
- Use `ReflectionBinding` for dynamic types that change at runtime
- Set `Background` colors appropriately:
  - DragCanvas: `White` (main workspace)
  - Toolbox area: Default or subtle color
  - DO NOT use bright colors (LightBlue, etc.) except for debugging

### C# Conventions
- Use CommunityToolkit.Mvvm attributes: `[ObservableProperty]`, `[RelayCommand]`
- Follow existing naming conventions for ViewModels (e.g., `StringSearchNodeViewModel`)
- Document complex methods with XML comments
- Use nullable reference types appropriately

## Project Structure

### Key Components
- **Views/**: XAML and code-behind files
- **ViewModels/**: MVVM ViewModels using CommunityToolkit.Mvvm
- **Nodes/**: RgxNode hierarchy (domain model)
- **Controls/**: Custom Avalonia controls (e.g., `RgxNodeControl`)
- **Avalonia.DragCanvas/**: Reusable drag-canvas library

### ViewModel Responsibilities
- `MainViewModel`: Top-level view state, commands, shared properties
  - `CurrentNodeViewModel`: The currently selected node's ViewModel
  - `SelectedCarouselIndex`: Which parameter panel to show
- Node-specific ViewModels: Wrap individual node types with bindable properties

## Common Pitfalls to Avoid

1. **Don't use Task.Delay() for timing**
   - This indicates a design flaw, not a solution
   - Find the proper event or lifecycle hook instead

2. **Don't break working drag-drop**
   - The toolbox uses direct TextBlock creation - this is intentional
   - Attempting to "fix" this with ItemsControl causes timing issues

3. **Don't set DataContext on individual controls**
   - Use a single parent DataContext and nested bindings
   - Example: `CurrentNodeViewModel` property bound by all carousel items

4. **Don't assume immediate visual tree realization**
   - ItemsControl containers may not exist immediately after ItemsSource changes
   - Use proper container events, not delays

## Incremental Changes
- Make ONE architectural change at a time
- Test thoroughly before proceeding to next change
- If something works, don't "improve" it without a specific need
- Revert quickly if a change breaks functionality

## Testing Checklist
Before committing MVVM or architectural changes:
- [ ] Toolbox drag-drop works
- [ ] Nodes can be placed on canvas
- [ ] Nodes can be connected via ports
- [ ] Parameter panels update when nodes are selected
- [ ] Node parameters are editable and persist to model
- [ ] Save/Load functionality works
- [ ] Visual layout is correct (no missing controls)
